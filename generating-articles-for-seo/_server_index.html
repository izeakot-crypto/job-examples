Документация

Полное описание архитектуры, компонентов и процессов сервиса STATI.

Обзор системы

STATI — внутренняя редакционная платформа Oki-Toki для автоматической генерации, верификации и публикации SEO-статей на 6 языках одновременно.

Как работают идеи

Идея — это тема будущей статьи. Все статьи начинаются с идеи. Идея проходит путь: создание → одобрение → генерация статьи.

Схема жизненного цикла идеи

Администратор нажимает "Генерировать идеи"
        │
        ▼
POST /api/ideas/generate  {count: 10}
        │
        ▼
Express: generateIdeas.js
  • Берёт темы из предустановленного списка (20 тем о call-центрах)
  • Случайно выбирает N тем и перемешивает
  • Для каждой темы:
    - генерирует UUID
    - извлекает ключевые слова из названия (фильтрует стоп-слова)
    - устанавливает priority_score (60–100, случайный)
    - status = "pending"
  • Сохраняет в таблицу ideas (Supabase)
        │
        ▼
Идеи появляются в разделе "Идеи" со статусом PENDING
        │
   ┌────┴────┐
   ▼         ▼
ОДОБРИТЬ   ОТКЛОНИТЬ
   │         │
   │         ▼
   │    status = "rejected"
   │    (можно восстановить)
   │
   ▼
POST /api/ideas/:id/approve
        │
        ▼
Express: ideasController.approveIdea
  • Меняет status = "approved" в Supabase
  • Делает HTTP POST → FastAPI :8000/pipeline/approve
        │
        ▼
FastAPI запускает WF1 (генерация статьи)
— см. раздел "Пайплайн"

Структура записи идеи (таблица ideas)

Поле Тип Описание
idUUIDУникальный идентификатор
titletextТема статьи (на украинском)
keywordstext[]Массив ключевых слов (используется в промпте)
statustextpending / approved / rejected
priority_scoreintПриоритет 60–100
descriptiontextКраткое описание темы
created_attimestamptzДата создания

Доступные действия с идеями

Одобрить status → approved, запускает пайплайн генерации. Только Admin.
Отклонить status → rejected, идея перемещается в "Отклонённые". Только Admin.
Восстановить Возвращает отклонённую идею в pending (через /approve). Только Admin.
Удалить Полное удаление из БД. Только Admin.

Пайплайн генерации статей

Запускается автоматически при одобрении идеи. Состоит из 3 воркфлоу (WF). Код: /python-pipeline/pipeline/

1
WF1 — Генерация
pipeline/generator.py · FastAPI endpoint: POST /pipeline/approve
· Языки генерируются последовательно (не параллельно) — Claude прокси принимает только 1 запрос одновременно
· Порядок: en → ru → pl → es → tr → uk
· Задержка 3 сек между языками
· При коротком результате — до 3 попыток на язык
· Если все попытки исчерпаны — принимает что есть (не блокирует)
· Результат передаётся напрямую в WF2 (не через БД)
2
WF2 — Верификация
pipeline/verifier.py · Вызывается WF1 автоматически
· Все 5 проверок запускаются параллельно через asyncio.gather()
· Если язык провалил → передаётся в WF3, потом WF2 повторяется
· Максимум 2 полных цикла верификации
· После WF2 сохраняет всё в Supabase (UPSERT) и уведомляет фронтенд
· Автоматически запускает генерацию изображения через Gemini
3
WF3 — Перегенерация
pipeline/regenerator.py · Вызывается WF2 при провале
· Получает точную причину провала: напр. "AI: 97% (>95% = fail); missing <ul>: found 1, need 2"
· Добавляет к промпту блок REGEN_INSTRUCTIONS с конкретными инструкциями
· Языки регенерируются последовательно, задержка 3 сек
· До 3 попыток на язык, возвращает лучший по длине

SSE события (real-time в браузер)

progress-update
Прогресс генерации. Поля: idea_id, stage, percent, message
stages: "generating"(30%) → "verifying"(70%) → "review_ready"(100%)
new-article
Статья готова. Поле article содержит полный объект — карточка появляется в списке
image-ready
Изображение сгенерировано. Поле article_image содержит base64 строку

Общие требования (все языки)

Формат ответаJSON: {{"SEO_HEADING": "...", "ARTICLE_HTML": "..."}}
Длина текстаМин. 12 000 символов без пробелов / макс. 18 000. При коротком → retry
СтруктураРовно 1×<h1>, 10–12×<h2>, под каждым 4–5×<p> по 50–80 слов
СпискиМин. 3 блока <ul><li> по 3–6 элементов (критично — проверяется WF2)
ВыделенияМин. 10×<strong> для ключевых терминов
БрендУпоминание Oki-Toki 5–6 раз естественно в тексте
Ключевое словоГлавный keyword — мин. 6 раз по всему тексту
Слова-связкиОднако, поэтому, более того, следовательно и т.п. — мин. 15 раз
Ссылки2–3 внешние ссылки на авторитетные источники (Wikipedia, отраслевые сайты)
Теги HTMLТолько: <h1> <h2> <h3> <p> <ul> <li> <strong> <a> — без <div> <span> <br>

Инструкции при регенерации (WF3)

К основному промпту добавляется блок с причиной провала. Пример для русского языка:

# Добавляется в конец промпта при регенерации
ВАЖНО — ИНСТРУКЦИИ ДЛЯ РЕГЕНЕРАЦИИ:
Предыдущая версия статьи НЕ прошла проверку качества.
Причины: {reason} # напр: "AI:97%;missing ul:1<2"
- Если 'water' высокий → меньше стоп-слов, пиши конкретнее
- Если 'nausea' высокий → больше разнообразия слов
- Если 'AI' высокий → разная длина предложений, уникальные примеры,
избегай клише "В современном мире" / "Стоит отметить"
- Если 'self-plagiarism' высокий → полностью оригинальный контент
Напиши СОВЕРШЕННО ДРУГУЮ статью — другая структура, другие примеры.

LLM клиент

· Основной клиент: anthropic.AsyncAnthropic с проксированием через локальный прокси
· Fallback: если задан OLLAMA_BASE_URL в .env — используется Ollama-совместимый клиент
· Таймаут: 300 сек для Claude, 600 сек для Ollama
· Прокси работает с concurrency limit: 1 (один запрос одновременно)

Верификация статей

Файл: pipeline/verifier.py. Каждый языковой вариант проходит 5 проверок параллельно.

Пороговые значения

Критерий             Провал (FAIL)        Предупреждение    Норма
─────────────────────────────────────────────────────────────────
AI-детекция          ≥ 95%                > 80%             < 80%
Водность             < 10% или > 50%      —                 10–50%
Акад. тошнота        < 0.3% или > 5%      —                 0.3–5%
Самоплагиат          > 25%               —                 ≤ 25%
Google уникальность  < 60%               —                 ≥ 60%
HTML <ul> списки   < 2 штуки           —                 ≥ 2
psychology
1. AI-детекция
services/verification/ai_detector.py
FAIL ≥ 95%
WARN > 80%
Анализирует plain text (HTML удаляется). Запрос к Claude через локальный прокси.
При провале — передаётся в WF3 с инструкцией: "разная длина предложений, уникальные примеры, избегай клише"
water_drop
2. Водность текста
services/verification/text_stats.py
FAIL <10% или >50%
OK 10–50%
Алгоритм: stop_words_count / total_words. Полностью локально, без внешних запросов.
Типичный результат хорошего текста: 15–40%. Стоп-слова — предлоги, союзы, частицы.
science
3. Академическая тошнота
services/verification/text_stats.py
FAIL <0.3% или >5%
OK 0.3–5%
Алгоритм: самое_частое_слово / все_смысловые_слова. Полностью локально.
Типичный диапазон: 0.5–3%. Высокая тошнота = слишком часто повторяется одно слово.
fingerprint
4. Самоплагиат
services/verification/self_plagiarism.py
FAIL > 25%
OK ≤ 25%
Shingle-сравнение: разбивает текст на N-граммы, сравнивает с уже существующими статьями того же языка в БД.
Предотвращает дублирование контента между разными статьями Oki-Toki.
travel_explore
5. Google уникальность
services/verification/google_checker.py
FAIL < 60%
OK ≥ 60%
Внешний сервис: Google Custom Search API. Проверяет насколько фрагменты статьи встречаются в индексе Google.
Требует: GOOGLE_CSE_KEY и GOOGLE_CSE_ID в .env файле пайплайна.
format_list_bulleted
6. HTML структура
verifier.py → _evaluate_thresholds()
FAIL < 2 тегов <ul>
OK ≥ 2
Regex: re.findall(r"<ul[\s>]", html). Полностью локально.
Требование явно указано во всех промптах как критическое. При провале — инструкция для Claude добавить списки.

Итоговый вердикт языка

fail — хотя бы 1 критерий провалился → язык идёт в WF3 на перегенерацию
warning — все прошли, но AI-детекция > 80% (предупреждение в UI)
pass — все 6 критериев в норме
skipped — язык не генерировался (нет контента)

API Reference

Все запросы (кроме /auth/login и /auth/verify) требуют заголовок x-auth-token: <token>. При 401 — автоматический logout.

Авторизация — публичные (без токена)
POST/auth/login
Body: {{"login":"email или логин","password":"..."}}
Response: {{"success":true,"token":"...","role":"admin|writer","name":"..."}}
GET/auth/verify
Header: x-auth-token · Response: {{"valid":true,"role":"...","login":"..."}}
Статьи /api/articles
GET/api/articles
Query: ?status=not_checked|checking|approved|published&offset=0&limit=12. Возвращает массив с полем translations:{uk,en,ru,pl,es,tr}
GET/api/articles/:id
Полные данные + верификация: zerogpt_*, uniqueness_*, verification_report, keywords, wordpress_url
PUT/api/articles/:id
Body: {{"status":"...","translations":{{"uk":{{"title":"","content":""}}}}}}
DELETE/api/articles/:id— только Admin
POST/api/articles/:id/publish
Публикация как черновик в WordPress. Требует WP_BASE_URL, WP_USERNAME, WP_APP_PASSWORD в .env
POST/api/articles/:id/generate-image
Генерация обложки через Gemini. Response: {{"image":"data:image/..."}}

Система токенов

Структура токена:
  <HMAC-SHA256-hex>.<BASE64(JSON_payload)>

Payload содержит:
  { id, login, role, ts(timestamp) }

Пример декодирования в браузере:
  JSON.parse(atob(token.split('.')[1]))

Хранение:
  localStorage  — при "Запомнить меня" (постоянно)
  sessionStorage — без "Запомнить" (до закрытия вкладки)

При 401 от API:
  → localStorage и sessionStorage очищаются
  → редирект на /login.html

Пользователи хранятся в:
  /var/www/seo-articles/api/users.json
  Пароли — bcrypt hash (salt rounds: 10)
  Вход по email ИЛИ по логину (оба варианта)
