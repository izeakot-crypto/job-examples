# Интеграция с Jira

## Как это работает

При деплое в PROD (merge в `main`) пайплайн автоматически проверяет коммит-сообщение.
Если в нём есть ключ задачи Jira (например `PROG-9537`), в эту задачу добавляется комментарий.

## Формат коммита

Укажи ключ задачи в начале первой строки коммита:

```
PROG-9537: техническое описание
```

### Простой коммит (только subject)

```
PROG-9537: добавлена интеграция с Jira
```

В Jira прилетит:
> добавлена интеграция с Jira
>
> Автор: Dmytro Intsin

### Коммит с бизнес-описанием (subject + body)

Если нужно отправить в Jira отдельное бизнес-описание — добавь его в тело коммита
(через пустую строку после первой строки):

```
PROG-9537: fix notify-jira - добавлен curl в зависимости

Настроена автоматическая отправка комментариев в задачи Jira
при деплое в продакшен. Теперь при каждом деплое задача
обновляется информацией о выполненной работе.
```

В Jira прилетит **только body** (бизнес-описание):
> Настроена автоматическая отправка комментариев в задачи Jira
> при деплое в продакшен. Теперь при каждом деплое задача
> обновляется информацией о выполненной работе.
>
> Автор: Dmytro Intsin

## Что произойдёт

1. **Ключ найден + есть body** — в Jira отправляется body (бизнес-описание) + автор
2. **Ключ найден + нет body** — в Jira отправляется текст коммита (без ключа) + автор
3. **Ключ не найден** — ничего не произойдёт, деплой пройдёт как обычно

## Когда срабатывает

- **Jira-комментарий** — только при деплое в **PROD** (ветка `main`)
- **Discord со ссылкой на Jira** — при деплое в PROD и TEST (ветки `main` и `staging`)
- На `feature/*` ветках ни Jira, ни Discord не срабатывают (только тесты)
- Шаг Jira идёт **после** деплоя — даже если Jira недоступна, деплой уже завершён

## Discord + Jira

При деплое в Discord автоматически приходит нотификация с полем **Задача** — кликабельная ссылка на Jira-задачу. Это работает и на staging (TEST), и на main (PROD).

**Jira-комментарий** отправляется только при деплое в PROD. В комментарий попадает body коммита (бизнес-описание) или subject (если body нет).

Итого при merge в main происходит:
1. Discord — embed с описанием коммита + ссылка на задачу
2. Jira — комментарий к задаче с бизнес-описанием + автор

## Полный процесс (git flow)

### 1. Коммит на feature-ветке

Простой коммит (subject = описание):
```bash
git commit -m "PROG-9537: добавлена валидация полей"
```

Коммит с бизнес-описанием (subject + body):
```bash
git commit -m "PROG-9537: добавлена валидация полей" -m "Реализована проверка обязательных полей формы. Теперь при отправке пустой формы показывается сообщение об ошибке."
```

### 2. Push feature — только тесты

```bash
git push origin feature/my-feature
```

### 3. Merge в staging — деплой TEST + Discord

```bash
git checkout staging && git merge feature/my-feature
git push origin staging
```

В Discord придёт нотификация:
- Container: **TEST**
- Описание: `PROG-9537: добавлена валидация полей`
- Задача: ссылка на PROG-9537

### 4. Merge в main — деплой PROD + Discord + Jira

```bash
git checkout main && git merge staging
git push origin main
```

В Discord придёт нотификация:
- Container: **PROD**
- Описание: `PROG-9537: добавлена валидация полей`
- Задача: ссылка на PROG-9537

В Jira (задача PROG-9537) появится комментарий:
> Реализована проверка обязательных полей формы. Теперь при отправке пустой формы показывается сообщение об ошибке.
>
> Автор: Dmytro Intsin

## Переменные в Bitbucket

В Repository Settings > Repository variables должны быть настроены:

| Переменная | Описание | Secured |
|---|---|---|
| `JIRA_URL` | URL Jira (например `https://jira.oki-toki.net`) | Нет |
| `JIRA_USER` | Логин пользователя Jira | Нет |
| `JIRA_PASSWORD` | Пароль пользователя Jira | **Да** |
| `DISCORD_DEPLOY_WEBHOOK` | Webhook для Discord-нотификаций о деплоях | **Да** |

## Поддерживаемые форматы ключей

Regex: `[A-Z]+-[0-9]+`

Любой ключ вида `БУКВЫ-ЦИФРЫ` будет распознан: `PROG-123`, `DEV-456`, `OKI-7890` и т.д.
Берётся первое совпадение в коммит-сообщении.
